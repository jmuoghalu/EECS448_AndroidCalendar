Code Freeze Date

All teams must have a code freeze as of 11:59pm September 18th. The timestamp will be judged by the final commit on your master branch of the team's github repository.

You can continue to work on other branches but cannot update your master branch after the freeze date. You must demo code base on that master branch as of the code freeze.

Overview

Pretty basic, make a calendar for this academic year (August 2016 through May 2017. You pick the platform and language.

Requirements

The user must be able do the following:

  • Set what the current date is
  • Choose 1 day display, 1 week display, 1 month display, 1 year display
    • 1 day displays a single date and any details that have been added to that day (discussed later)
    • 1 week display shows the Sunday to Saturday dates surrounding whatever the date currently is
  • Be able to add details to particular date
    • Example: I can add the entry "No class" to October 10th, then when I look at that day in 1-day mode I see the detail I added
    • Details are only visible when in 1 day display mode
  • The calendar must able to add details and remove details from a date
  • The details must be retained between runnings of the program
    • Nope, we won't have the database lab in time, but luckily we all learned file I/O!

Some aspects that are up to you to decide on:

  • Menus to interact the calendar
  • How to deal with the user across multiple sessions
  • Again, language and platform

Team Meetings

You will have the lab session of the week of 9/12 for team meetings. Meetings are to take place in lab. If you plan/prefer to meet somewhere email your TA and let them know you have a meeting room elsewhere.

Plagiarism

You can use existing libraries (e.g. timing libraries), but you must site all sources of code you do no generation in your documentation.

Team Score (70% of individual grade)

This portion of the project will be judged by myself and the TA. The project points are broken down into the following sections.

Presentation

Each team will give a presentation in class to go over the development process and demo their product. Topics that must be discussed in the presentation:

  • Overview of the platform
  • Description on how work was split between teammates
  • Challenges and how they were overcome or dealt with
  • Any features that did not make the demo version
  • Retrospective on what the team would have done different
  • Be professional
    • No need to dress up, but you need present yourself and your work in a professional manner

Modularity

In short, your code should be:

  • object oriented (I got my eye on you JavaScript people)
  • easily extensible
  • divided into logical components (i.e. not one class that does everything)

More details in lectures.

Stability

You demo the code in class and the TA and myself will stress test your application at our leisure. Crashes, memory leaks, or other things that you also hate in bad software will be met with penalty.

User Interface

Your product should be intuitive to use. How you accomplish this is up to you. Good rule to stick by: If I need a manual to use your interface, you have a bad interface.

Code Documentation

You must use some kind of documentation format/software, such as JavaDoc from EECS 268, to provide documentation of your code. In short, you need to use some means of documentation that can be used to generate HTML documentation of your code. We should be able to open HTML file to find out all the class information such as method names, signatures, pre and post conditions, etc.

On your github repository, have a folder called "documentation" that contains all the generated documentation, readmes, work cited list, and any other needed documentation.

Works cited section

This will be it's own section in the documentation. Any code that you or a teammate did not write from scratch, must be cited. This should be on your repository also.

Team Evaluations (20% of individual grade)

You will evaluate all of your teammates. You will only share the evaluations with me and our TA. The teammates your evaluating will not have access to these evaluations.

The score one received for this component is an average of all their teammates evaluations. In other words, your teammates evaluation of you will affect your grade for this project.


Link to self evals coming after code freeze date.

Self evaluations (10% of individual grade)

You will evaluate your own performance on the project. You'll have a chance to list your own struggles and successes and give yourself an overall score.
